docker-compose.yml
  Defines 3 services that run together:
  - mongo:    Official MongoDB 7 image. Stores all data in a named volume
              so data survives container restarts.
  - backend:  Builds from backend/Dockerfile. Mounts ./backend as a volume
              so code changes reload instantly (uvicorn --reload).
              Depends on mongo so it starts after the database is ready.
  - frontend: Uses node:20-alpine, runs npm install + npm run dev.
              Depends on backend. Exposes port 5173.

README.md
  Quick-start instructions and architecture summary for anyone who
  opens the GitHub repo.

.gitignore
  Prevents committing node_modules, Python venv, .env secrets,
  __pycache__, and build artifacts.

requirements.txt
  All Python dependencies pinned to exact versions:
  - fastapi + uvicorn   → web framework and ASGI server
  - motor + beanie      → async MongoDB driver + ODM (Object Document Mapper)
  - python-jose         → JWT token creation and verification
  - passlib[bcrypt]     → password hashing
  - python-multipart    → required for file upload (multipart/form-data)
  - pydantic-settings   → reads .env file into typed Settings class
  - PyMuPDF (fitz)      → extracts text from PDF files
  - python-docx         → extracts text from DOCX files
  - sentence-transformers → AI model for semantic similarity scoring
  - scikit-learn        → cosine_similarity calculation
  - spacy               → NLP library (loaded but skill extraction uses
                          regex + predefined list for speed)

Dockerfile
  Builds the backend image:
  1. Starts from python:3.11-slim (small base image)
  2. Installs all requirements
  3. Downloads spaCy English model (en_core_web_sm)
  4. Copies app code
  5. Starts uvicorn with --reload for development

The entry point for the entire backend.

  - Creates the FastAPI app instance
  - Adds CORS middleware so the React frontend (port 5173) can call the API
  - On startup: connects to MongoDB and initializes Beanie with all
    document models (User, Resume, JobDescription, CandidateMatch)
  - Registers the 3 API routers: auth, resumes, jobs
  - Exposes GET /health for uptime checks

Why Beanie?
  Beanie is an async ODM built on Motor (async MongoDB driver) and Pydantic.
  It lets you define MongoDB collections as Python classes with type hints,
  and provides async CRUD methods like .insert(), .find(), .get().

All authentication and authorization logic lives here.

  hash_password(password)
    Hashes a plain-text password with bcrypt before storing in MongoDB.

  verify_password(plain, hashed)
    Checks a login attempt against the stored bcrypt hash.

  create_access_token(data)
    Creates a signed JWT containing the user's ID and role.
    Expires after ACCESS_TOKEN_EXPIRE_MINUTES (default 24h).

  get_current_user (FastAPI Dependency)
    Reads the Bearer token from the Authorization header,
    decodes it, and returns the User document from MongoDB.
    Used with Depends() on any protected route.

  require_role(*roles) (FastAPI Dependency factory)
    Wraps get_current_user and additionally checks the user's role.
    Usage: Depends(require_role("seeker")) or Depends(require_role("recruiter"))
    Returns 403 Forbidden if the role doesn't match.


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
BACKEND — app/models/user.py
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Defines all 4 MongoDB collections as Beanie Document classes.

  User
    Stores registered accounts.
    Fields: email, hashed_password, role ("seeker"/"recruiter"),
            full_name, created_at
    Collection: "users"

  Resume
    Stores an uploaded resume and its extracted data.
    Fields: user_id (links to User), filename, raw_text (full extracted text),
            skills (list of detected skills), ats_score (set after matching)
    Collection: "resumes"

  JobDescription
    Stores a job posting created by a recruiter.
    Fields: recruiter_id (links to User), title, description,
            required_skills (auto-extracted from description text)
    Collection: "job_descriptions"

  CandidateMatch
    Stores the result of matching one resume against one job.
    Fields: jd_id, resume_id, user_id, score (0-100), matched_skills,
            missing_skills, explanation (human-readable string)
    Collection: "candidate_matches"

All models use PydanticObjectId for MongoDB _id fields, which
automatically serializes to/from string in JSON responses.


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
BACKEND — app/services/nlp.py
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

The AI brain of the application. All NLP and scoring logic.

  get_model()
    Lazy-loads the SentenceTransformer model "all-MiniLM-L6-v2".
    Called only when first needed, then cached in a module-level variable.
    This avoids reloading the 80 MB model on every request.

  SKILLS_DB (set)
    A curated set of ~60 skills covering tech (Python, Docker, AWS, React...)
    and non-tech (leadership, Excel, Figma...). Used for keyword matching.
    Can be extended freely — just add strings to the set.

  extract_text_from_file(content, filename)
    Reads raw bytes of an uploaded file.
    - PDF: uses PyMuPDF (fitz) to extract text from all pages
    - DOCX: uses python-docx to join all paragraph texts
    - Other: raises ValueError (caught by the API and returned as 400)

  extract_skills(text)
    Lowercases the text, then checks each skill in SKILLS_DB using
    regex word-boundary matching (\b) to avoid false positives
    (e.g., "C" matching inside "science").
    Returns a sorted, deduplicated list.

  compute_ats_score(resume_text, jd_text)
    Encodes both texts into embedding vectors using sentence-transformers.
    Computes cosine similarity between the two vectors.
    Returns a score from 0 to 100 representing semantic closeness.

  match_resume_to_jd(resume_skills, jd_skills, resume_text, jd_text)
    The main matching function. Combines two signals:
    1. Skill overlap score: matched_skills / total_jd_skills × 100
    2. Semantic score: compute_ats_score() result
    Final score = 0.6 × semantic + 0.4 × skill_overlap
    Also builds the explanation string shown in the UI.
    Returns: { score, matched_skills, missing_skills, explanation }

Why combine both signals?
    Semantic similarity catches cases where the resume uses different
    words for the same concept (e.g., "ML" vs "machine learning").
    Skill overlap catches exact keyword matches that recruiters care about.

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Handles user registration and login. Prefix: /api/auth

  POST /api/auth/register
    Accepts: { email, password, full_name, role }
    Validates role is "seeker" or "recruiter".
    Checks email is not already registered.
    Hashes password and saves User to MongoDB.
    Returns: { id, email, role }

  POST /api/auth/login
    Accepts: OAuth2 form data (username=email, password)
    Verifies credentials against stored bcrypt hash.
    Returns: { access_token, token_type, role, full_name }
    The frontend stores this token in localStorage and sends it
    as "Authorization: Bearer <token>" on every subsequent request.

BACKEND — app/api/resumes.py
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Handles resume upload and retrieval. Prefix: /api/resumes
All routes require authentication. Upload requires role="seeker".

  POST /api/resumes/upload
    Accepts: multipart file upload (.pdf or .docx)
    Extracts text → extracts skills → saves Resume document.
    Returns: { id, skills, filename }

  GET /api/resumes/mine
    Returns all resumes belonging to the logged-in seeker.
    Includes filename, skills list, and ats_score (if matched).

  GET /api/resumes/{resume_id}
    Returns a single resume. Seekers can only access their own.
    Recruiters can access any resume (for viewing ranked results).


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
BACKEND — app/api/jobs.py
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Handles job descriptions and candidate ranking. Prefix: /api/jobs

  POST /api/jobs/
    Recruiter posts a job. Auto-extracts required_skills from description.
    Returns: { id, required_skills }

  GET /api/jobs/
    Returns all posted jobs. Accessible by both roles.
    Seekers use this to browse jobs and trigger matching.

  POST /api/jobs/{jd_id}/rank
    Recruiter uploads multiple resumes at once.
    For each file: extracts text → extracts skills → runs match_resume_to_jd
    → saves Resume + CandidateMatch documents.
    Returns results sorted by score descending.
    This is the core recruiter workflow.

  GET /api/jobs/{jd_id}/matches
    Returns previously saved match results for a job, sorted by score.
    Useful for revisiting rankings without re-uploading.

  GET /api/jobs/{jd_id}/match-my-resume
    Seeker matches their own uploaded resume against a specific job.
    Query param: resume_id
    Updates the resume's ats_score field in MongoDB.
    Returns: { score, matched_skills, missing_skills, explanation }

  as PostCSS plugins during the build.

frontend/Dockerfile
  Multi-stage build for production:

  Stage 2: Copies dist/ into nginx:alpine to serve static files.

Zustand store for global auth state.
Persists token, role, and fullName to localStorage so the user
stays logged in after a page refresh.

  login(token, role, fullName)   → saves to state + localStorage
  logout()                       → clears state + localStorage

Any component can read auth state with:
  const { token, role } = useAuthStore()

The most feature-rich seeker page. Shows all available jobs and
lets the seeker match their resume against any of them.

On mount: fetches all jobs + all of the seeker's resumes.
Resume selector: dropdown to choose which resume to use for matching.

For each job card:
  - Shows job title and required skills
  - "Match" button calls GET /api/jobs/{jd_id}/match-my-resume
  - After matching, expands to show:
      * RadialBar chart (recharts) visualizing the score percentage
      * Matched skills (green badges)
      * Missing skills (red badges)
      * Explanation text from the AI

The core recruiter tool. Upload multiple resumes and get them
ranked against a selected job.

  Job selector: dropdown of all posted jobs
  File input:   accepts multiple .pdf/.docx files at once
  On submit:    POST /api/jobs/{jd_id}/rank with all files

Results display (sorted by score, highest first):
  - Rank number (#1, #2, ...)
  - Filename
  - Score with color coding:
      green  ≥ 70%  (strong match)
      yellow ≥ 40%  (partial match)
      red    < 40%  (weak match)
  - Matched skills (green badges)
  - Missing skills (red badges)
  - AI explanation text (italic, below a divider)

This is the "Explainable AI" feature — recruiters see exactly
WHY a candidate ranked where they did, not just a number.

DATA FLOW SUMMARY
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Seeker uploads resume:
  Browser → POST /api/resumes/upload
  → extract_text_from_file() → extract_skills()
  → Resume saved to MongoDB
  → skills returned to UI

Seeker matches resume to job:
  Browser → GET /api/jobs/{jd_id}/match-my-resume?resume_id=...
  → load Resume + JobDescription from MongoDB
  → match_resume_to_jd() → semantic + skill scoring
  → ats_score updated on Resume document
  → score + explanation returned to UI

Recruiter ranks candidates:
  Browser → POST /api/jobs/{jd_id}/rank (multiple files)
  → for each file: extract text → extract skills → match_resume_to_jd()
  → Resume + CandidateMatch saved to MongoDB
  → sorted results returned to UI
